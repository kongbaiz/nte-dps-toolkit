# NTE 伤害解析维护摘要

本文是公开仓库中的降敏维护摘要，只描述解析模块的职责边界、输入输出语义和保守策略。可直接复现实验的抓包样本、精确特征字节、客户端函数名、资源导出路径、完整载荷、PCAP 文件名和偏移细节不应随公开仓库发布。

解析逻辑的权威实现位于 `src/protocol.rs`、`src/parser.rs`、`src/capture.rs` 和 `src/model.rs`。修改解析规则时，以代码和测试为准，并在提交说明中避免记录敏感样本内容。

## 设计边界

- `protocol.rs` 负责 UE 传输层位流解析，保持纯函数、确定性、无文件系统、无时间、无 UI。
- `parser.rs` 负责伤害载荷、GameplayEffect、技能分类和资源表读取，解析失败返回 `Option` 或 `Result`，不得 panic。
- `capture.rs` 负责实时抓包、PCAPNG/JSON 导入、原始帧写入和 `EngineEvent` 产生，不依赖 UI。
- `model.rs` 负责领域模型、导出结构、战斗聚合和深渊状态，不依赖 UI 或 Win32 API。

实时抓包、PCAPNG 回放和 JSON 回放应尽量复用同一条稳定解析流程，避免某个入口独享额外规则导致统计语义漂移。

## 输入来源

程序只处理用户本机运行时可见的数据：

- 实时 UDP 流量。
- Debug 保存的 PCAPNG。
- 程序导出的 JSON。
- 仓库内稳定资源表。

公开文档、Issue 和提交说明中不要粘贴完整载荷、抓包片段、资源导出密钥、usmap、本机路径或可直接还原客户端内部实现的定位信息。

## 解析原则

解析器遵循“宁可不识别，也不要制造高置信误判”的原则：

- 所有长度、位偏移、数组访问和数值转换必须先做边界检查。
- 伤害记录需要同时满足字段结构、数值范围、重复字段一致性和方向判定条件。
- GameplayEffect 归因必须能从同包上下文或稳定资源表中得到足够证据。
- 低置信度候选只进入 Debug 备注或暂存队列，不直接进入统计。
- 服务端 HP 同步和客户端伤害记录之间的关联必须有时间窗、方向和目标状态约束。

新增启发式时至少覆盖正常样本、边界样本和误判规避样本。没有样本覆盖的规则只应作为 Debug 观察项。

## 方向和归属

方向判定优先使用本机游戏 IP 与端点信息；无法确定时再使用私网/公网关系、角色声明和已确认客户端端点作为回退。

角色归属优先使用同包同位流上下文中的角色声明；无法消歧时使用同一 UDP 四元组最近一次确认角色。多角色上下文中无法确认的候选应暂存或标记为未知，不能直接归入某个角色。

稳定主线不主动填充或显示敌方目标识别字段。`target_id`、`target_name`、`target_context` 只作为旧 JSON 兼容字段保留。

## 伤害和技能分类

每条命中会尽量补全以下信息：

- 基础伤害、目标 HP 前后值和目标最大 HP。
- 方向：输出、受击或未知。
- 角色 ID 与角色显示信息。
- GameplayEffect 索引和名称。
- 技能大类、具体招式名和伤害属性。
- 深渊上下行线状态。

技能分类优先使用 `res/data/skills/` 下的稳定资源表。资源表缺项时，可以使用名称约定做保守降级；仍无法判断时显示“其他”或“未知”。

## 环合和追加伤害

环合类伤害应优先通过稳定 GameplayEffect 或明确的上下文证据识别。追加伤害不应随意生成独立命中；如果它属于某条原始命中的后续结算，应通过 `follow_up_damage` 合并回源命中，并保证总览、角色统计、技能汇总和深渊统计都使用 `Hit.total_damage()`。

覆纹等依赖服务端 HP 差值的规则必须保持保守：

- 只在已确认激活上下文后记录。
- 按发生顺序关联原始伤害和 HP 更新。
- 避免跨战斗、跨目标或跨长时间间隔复用旧队列。
- 不用单个数值相近作为归因依据。

## JSON 和兼容性

导出结构新增字段必须使用 `#[serde(default)]` 或提供旧 JSON 可导入的兼容行为。

导入旧 JSON 时：

- 缺失方向字段默认按旧语义兼容。
- 旧攻击类型名称在导入时迁移到当前 UI 术语。
- 旧目标识别字段仅保留，不在稳定 UI 主动展示。

## Debug 数据

Debug 数据用于本机排查，不是公开报告材料。程序可以保留原始帧、解析备注和候选信息，但生成文件必须位于忽略目录，提交和报告中只保留脱敏统计结论。

如果需要说明一次修复的依据，优先写：

- 触发场景的抽象描述。
- 解析前后的统计差异。
- 已补充的测试覆盖。
- 仍然保守不识别的边界。

避免写：

- 抓包文件名。
- 完整十六进制片段。
- 客户端符号、函数名或偏移。
- 资源导出密钥、usmap、客户端安装路径。
- 可直接复现的协议特征组合。

## 维护检查表

修改解析或统计逻辑后至少检查：

- `cargo fmt --check`
- `cargo check`
- `cargo test`

涉及真实抓包的诊断测试默认忽略，只在本机具备合法样本时运行，并且不得把样本或本机路径写入提交说明。

涉及 GameplayEffect、技能分类、伤害属性或深渊事件时，同步检查：

- `res/data/skills/`
- `res/data/reactions/`
- `res/data/abyss/`
- 相关 UI 汇总和 JSON 导入导出字段
